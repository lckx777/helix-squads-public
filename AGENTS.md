# AGENTS.md — Helix Squads Public Marketplace

Maintenance guide for the marketplace repo. Any contributor (or the maintainer's
agents) should read this before modifying the repo.

## Core rules

1. The source of truth is `~/.claude/skills/squads/squads/<squad>/` on the
   maintainer's machine. Plugins here are generated from that SoT by
   `scripts/convert-squad-to-plugin.py`.
2. Never edit `plugins/<name>/` files directly — they get overwritten by the
   next sync. Edit the SoT and re-run conversion.
3. `marketplace.json` is auto-generated by `scripts/build-marketplace.py`.
   Never hand-edit the `plugins` array. Edit the top-level metadata only.
4. Every commit on `main` is validated by CI. PRs only.
5. All commits are SSH-signed (see LOCAL-SETUP.md in the private repo for the
   maintainer's one-time GitHub configuration).

## Adding a new plugin (typical flow)

```
# 1. Ensure the squad exists at ~/.claude/skills/squads/squads/<name>
# 2. Convert it into a plugin
python3 scripts/convert-squad-to-plugin.py <name> plugins

# 3. Rebuild marketplace.json
python3 scripts/build-marketplace.py

# 4. Validate
python3 scripts/validate-plugins.py

# 5. Commit and PR
git checkout -b add-<name>
git add plugins/<name> .claude-plugin/marketplace.json
git commit -m "feat: add <name> plugin"
git push -u origin add-<name>
gh pr create --fill
```

## Removing a plugin

```
rm -rf plugins/<name>
python3 scripts/build-marketplace.py
python3 scripts/validate-plugins.py
git add -A
git commit -m "feat: remove <name> plugin"
```

## Release

Tag the repo with semver:

```
git tag -s v1.0.0 -m "v1.0.0 — <release notes>"
git push origin v1.0.0
```

CI creates a GitHub Release automatically.

## Updating an existing plugin

The SoT squads may drift. To re-sync a single plugin:

```
# Delete the current plugin dir (or script refuses to overwrite)
rm -rf plugins/<name>

# Re-convert
python3 scripts/convert-squad-to-plugin.py <name> plugins

# Rebuild + validate
python3 scripts/build-marketplace.py
python3 scripts/validate-plugins.py

# Bump version in plugin.json if breaking
# (convert script reads version from squad.yaml by default)
```

## Plugin format reference

Each plugin directory:

```
plugins/<name>/
├── .claude-plugin/
│   └── plugin.json            # manifest (required)
├── agents/                    # persona markdown files with frontmatter
├── skills/                    # each subdir has SKILL.md (model-invoked)
├── commands/                  # flat .md files (user-invoked slash commands)
├── workflows/                 # YAML pipelines (read by skills at runtime)
├── chains/                    # YAML chains (raw, preserved for reference)
├── references/                # static knowledge: checklists, config, data
├── scripts/                   # helper shell scripts
├── squad.yaml                 # preserved from SoT for legacy consumers
└── README.md
```

Claude Code namespaces skills as `/<plugin-name>:<skill-name>` to prevent
conflicts. Agents retain their original names inside the plugin.

## CI

- `.github/workflows/validate.yml` runs on every PR:
  - `validate-plugins.py` checks structure
  - `build-marketplace.py` is run and the result compared against the
    committed `marketplace.json` — if they differ, CI fails (forces
    contributor to commit the rebuilt file)
  - `gitleaks` scans for secrets
- `.github/workflows/release.yml` runs on tag push — creates a GitHub Release

## Hermes agent notes

If an AI agent is modifying this repo:
- Respect the "SoT first, plugins generated" rule. Do not hand-edit plugins.
- Run `scripts/validate-plugins.py` before every commit; fail fast locally
  rather than in CI.
- Sign commits (SSH signing is already configured globally for this user).
- Do not delete plugins without explicit user authorization — even if they
  look stale.
